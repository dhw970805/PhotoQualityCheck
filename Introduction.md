# 人像摄影图片质量检测工具

## 项目简述
本项目旨在利用大模型辅助婚纱摄影工作室进行原图初选。系统将结合传统计算机视觉（MediaPipe）与多模态大模型（Qwen-3.5Plus），快速筛选出存在严重质量问题（如闭眼、表情僵硬、构图失误、曝光异常）的废片，并为优质照片提供评分。

## 技术栈
*   **前端**：React + Material UI (深色主题)
*   **后端**：Python + Flask
*   **数据库/存储**：Chroma (只写框架,暂时不实现数据库的功能)
*   **视觉分析**：MediaPipe, Qwen-3.5Plus (大模型)

---

## 核心工作流设计

### 1. 前端工作流
1.  **文件选择**：用户点击选择本地文件夹，前端展示文件夹内所有图片缩略图。
2.  **状态初始化**：在文件夹下建立/读取 `result.json`，为每张图片初始化数据结构，状态默认为 `未检测`。
3.  **触发筛选**：用户点击“开始筛选”，前端将图片路径列表发送给后端。
4.  **实时反馈**：页面核心区域展示照片，并根据 `result.json` 实时更新照片上的状态标签（如“闭眼”、“合格”）。
5.  **结果导出**：用户点击“导出”，根据筛选结果将图片移动到对应的子文件夹（如 `废片` / `待修`）。

### 2. 后端工作流 (关键逻辑)
后端流程必须严格遵循以下顺序，以确保效率和准确性：

1.  **增量扫描**：读取 `result.json`，仅处理状态为 `未检测` 的图片。
2.  **规则引擎 (MediaPipe)**：
    *   分析人脸特征、五官位置。
    *   **判定逻辑**：
        *   若无人脸 -> 直接标记为 `需复核`，**跳过**大模型推理。
        *   若通过规则筛选 -> 进入大模型推理。
3.  **大模型推理 (Qwen-3.5Plus)**：
    *   仅发送通过 MediaPipe 筛选的图片。
    *   **容错机制**：若大模型返回非 JSON 格式或超时，需捕获异常，标记为 `未检测` 并记录日志，**不能阻塞**后续图片的处理。
4.  **结果聚合**：将模型结果写入 `result.json`，更新状态。

## 数据结构设计

### 1. `result.json` 结构
```json
{
  "photos": [
    {
      "photo_metadata": {
        "file_info": {
          "file_name": "DHCF0011.JPG",
          "file_path": "D:\\backup\\DHCF0011.JPG",
          "hash": "sha256_value"
        },
        "quality": [
          "闭眼",
          "欠曝",
        ],
        "status": "需复核", // 枚举值: "未检测", "合格", "需复核", 
        "scores": {
          "expression": 50,
          "composition": 70,
          "exposure": 40
        },
        "advise": "新娘表情不自然，建议重拍",
        "reason": "检测到主要人物闭眼且无笑意"
      }
    }
  ]
}
```

### 2. `quality` 类型枚举
*   `未检测` (初始状态)
*   `闭眼`
*   `表情差`
*   `构图差`
*   `欠曝`
*   `过曝`
*   `合格`

## 大模型 Prompt 设计 (Qwen-3.5Plus)

**Role**: 资深婚庆摄影修图师和选片专家 (拥有 20 年经验)。

**Goal**: 快速从大量婚礼照片中筛选出“废片”。摄影师拍摄约 3000 张，你需要剔除严重质量问题照片。请采用“商业交付级”标准，降低宽容度。

### Critical Definitions (判定标准)
满足任意一条即视为“坏照片”：
1.  **表情不自然 (Unnatural Expression)**:
    *   **判定**：嘴巴微张/歪斜（说话中）、眼神空洞/失焦、面部肌肉用力过渡（假笑酝酿）、抿嘴过紧。
    *   **自检**：如果新人看到这个表情会觉得尴尬，则判为坏照片。
2.  **主要人物闭眼 (Blinking)**:
    *   **判定**：主要人物双眼闭合，面部松弛。
    *   **例外**：**必须严格区分**。如果是开怀大笑导致的闭眼（眼角有笑纹，嘴角上扬），则是好照片。
3.  **构图不好 (Poor Composition)**:
    *   **判定**：主要人物被遮挡（脸部被挡）、切头切脚、画面严重倾斜、背景杂乱干扰主体。
4.  **曝光问题 (Exposure Issues)**:
    *   **欠曝**：人物面部暗沉发黑，强行提亮会噪点爆炸。
    *   **过曝**：高光溢出（如婚纱死白），面部无细节，肤色惨白。

### Output Format (严格遵守)
**注意**：`reasons` 数组中的字符串必须严格使用中文标签，且必须是 `quality` 枚举值中的内容。
`quality` 类型枚举
*   `闭眼`
*   `表情差`
*   `构图差`
*   `欠曝`
*   `过曝`
*   `合格`
*   `需复核`

```json
{
  "is_bad_photo": true,
  "expressionScore": 50,
  "compositionScore": 60,
  "exposureScore": 40,
  "reasons": [
    "闭眼",
    "欠曝"
  ],
  "detailed_analysis": "新娘在严肃时刻意外眨眼，且背景光线不足导致面部发灰"
}
```

### Constraints
1.  **输出必须是纯 JSON**，不要包含任何 Markdown 代码块标记（如 \`\`\`json）。
2.  **只关注主要人物**（新郎/新娘），忽略背景路人。
3.  **容错原则**：如果图片模糊无法判断，或者模型无法给出确定结论，请输出需复核，防止误删珍贵瞬间。

## 前端界面设计

### 1. 整体风格
*   **主题**：深色模式 (Dark Mode)。
*   **配色**：背景使用 Material UI 默认深灰 (`#121212`)，卡片使用 (`#1E1E1E`)，文字使用白色或浅灰色。
*   **布局**：
    *   **顶部**：菜单栏 (选择文件夹, 导出分类)。
    *   **中部**：核心展示区 (网格布局展示图片)。
    *   **右侧**：详情面板 (展示选中照片的评分和详细分析, 用户可编辑照片检测结果和详细信息)。
    *   **底部**：状态栏 (进度条, "开始检测" 按钮, 取消按钮)。

### 2. 交互细节
*   **状态标识**：每张图片上需叠加状态标签。
    *   `未检测`：灰色圆点。
    *   `合格`：绿色对勾。
    *   `需复核` (含废片标签)：红色感叹号。
    *   result.json中quality字段的内容,每个状态以不同的颜色展示
*   **实时更新**：后端每处理完一张图片，前端需立即刷新该图片的状态，无需刷新页面。

### 3. 监测与日志
*   建立日志机制，记录 Token 消耗、请求耗时、错误信息。
*   若大模型请求超时或报错，前端需在详情面板显示错误信息，并允许用户对该图片进行“重试”。
